# 逻辑原型

一个小型交互式终端 app，让用户手动驱动状态模型。当问题涉及**业务逻辑、状态转换或数据形状**时使用它——这类问题在纸面上看起来合理，但只有拿真实案例推一遍，才会暴露哪里不对。

## 什么时候适合这种形态

- “我不确定这个状态机能不能处理先 X 后 Y 的边界情况。”
- “这个数据模型真的能表示……这种情况吗？”
- “我想在写 API 前先感受一下 API 应该是什么形状。”
- 任何用户想要**按键并观察状态变化**的场景。

如果问题是“这应该长什么样”——那就是选错分支。使用 [UI.md](UI.md)。

## 流程

### 1. 写明问题

写代码前，先写下你要原型化的状态模型和问题。可以是原型 README 里的一段话，也可以是文件顶部注释。回答错问题的逻辑原型纯属浪费——把问题写清楚，方便之后检查，无论用户现在是否在看，还是稍后 AFK 回来。

### 2. 选择语言

使用宿主项目已经在用的语言。如果项目没有明显运行时（例如文档仓库），就询问用户。

匹配项目现有工具约定——不要只为原型新增包管理器或运行时。

### 3. 把逻辑隔离到可移植模块中

把真正的逻辑——回答问题的那部分——放在一个小而纯粹的接口后面，之后可以直接取出并放进真实代码库。外层 TUI 是一次性的；逻辑模块不应该是。

正确形态取决于问题：

- **纯 reducer** — `(state, action) => state`。当 action 是离散事件、state 是单一值时很适合。
- **状态机** — 显式状态和转换。当“当前到底允许哪些动作”本身也是问题的一部分时很适合。
- **围绕普通数据类型的一小组纯函数**。当没有隐式当前状态，只有转换时很适合。
- **拥有清晰方法表面的类或模块**，用于逻辑确实拥有持续内部状态的情况。

选择最贴合问题的形态，*不要*选择最容易接到 TUI 上的形态。保持纯粹：没有 I/O、没有终端代码、没有用 `console.log` 控制流程。TUI 导入并调用它；不要让依赖反向流动。

这就是原型在结束后仍然有价值的原因。问题回答完后，验证过的 reducer / machine / function set 可以提升到真实模块中——TUI 外壳则删除。

### 4. 构建能暴露状态的最小 TUI

把它做成**轻量级 TUI**——每一轮都清屏（`console.clear()` / `print("\033[2J\033[H")` / 等价方式），并重新渲染整帧。用户应该始终看到一个稳定视图，而不是不断增长的滚动日志。

每一帧按顺序包含两部分：

1. **当前状态**，美观打印，并且便于 diff（每行一个字段，或格式化 JSON）。字段名或小节标题使用**粗体**，不太重要的上下文（时间戳、ID、派生值）使用**暗色**。原生 ANSI 转义码就够了——`\x1b[1m` 粗体，`\x1b[2m` 暗色，`\x1b[0m` 重置。除非项目已经有样式库，否则不用引入。
2. **键盘快捷键**，列在底部：`[a] add user  [d] delete user  [t] tick clock  [q] quit`。可以把按键加粗、描述置暗，也可以反过来——只要读起来清楚。

行为：

1. **初始化状态** — 一个内存中的对象/结构。启动时渲染第一帧。
2. **一次读取一个按键（或一行）**，分派给会改变状态的 handler。
3. **每次操作后重新渲染**完整帧——不要追加，直接替换。
4. **循环直到退出。**

整帧应该能放进一个屏幕。

### 5. 让它用一个命令运行

把脚本加入项目现有任务运行器（`package.json` scripts、`Makefile`、`justfile`、`pyproject.toml`）。用户应该能运行 `pnpm run <prototype-name>` 或等价命令——不需要记住路径。

如果宿主项目没有任务运行器，就把命令放在原型 README 顶部。

### 6. 交接

把运行命令给用户。他们会自己操作；真正有价值的时刻是他们说“等等，这不应该发生”或“咦，我以为 X 会不一样”——这些就是_想法_里的 bug，也正是原型的意义。如果他们想新增操作，就继续添加。原型会演进。

### 7. 捕获答案

当原型完成使命后，问题的答案是唯一值得保留的内容。如果用户在线，问问它让他们学到了什么。如果不在线，就在原型旁边留下 `NOTES.md`，让答案能在原型删除前补上（如果你看过这次会话，也可以由你补）。

## 反模式

- **不要添加测试。** 需要测试的原型已经不再是原型。
- **不要接入真实数据库。** 除非问题专门关于持久化，否则使用内存存储。
- **不要泛化。** 不要想“如果以后要支持 X 怎么办”。原型只回答一个问题。
- **不要混淆逻辑和 TUI。** 如果 reducer / state machine 引用了 `console.log`、prompt 或终端转义码，它就不再可移植。让 TUI 成为纯模块外的一层薄壳。
- **不要把 TUI 外壳发到生产。** 这个外壳是为人在终端里手动驱动而优化的。它背后的逻辑模块才是值得保留的部分。
