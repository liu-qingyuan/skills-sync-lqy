# 设计两次

当用户想为某个已选定的深化候选项探索替代接口时，使用这个并行 sub-agent 模式。它基于 Ousterhout 的 “Design It Twice” 思路——第一个想法通常不会是最好的。

本文使用 [SKILL.md](SKILL.md) 中的词汇 — **模块**、**接口**、**接缝**、**适配器**、**杠杆**。

## 流程

### 1. 界定问题空间

在启动 sub-agent 之前，先为所选候选项写一段面向用户的问题空间说明：

- 任何新接口都必须满足的约束
- 它会依赖哪些东西，以及这些依赖属于哪一类（见 [DEEPENING.md](DEEPENING.md)）
- 一段粗略的示意代码，用来把约束落到具体场景里——这不是方案，只是让约束更清楚

把这段说明展示给用户，然后立刻进入步骤 2。用户可以一边阅读和思考，一边等待 sub-agent 并行工作。

### 2. 启动 sub-agent

使用 Agent 工具并行启动 3 个以上 sub-agent。每个 sub-agent 都必须为深化后的模块产出一个**明显不同**的接口。

给每个 sub-agent 一份独立的技术简报（文件路径、耦合细节、来自 [DEEPENING.md](DEEPENING.md) 的依赖类别、接缝背后放什么）。这份简报独立于步骤 1 中面向用户的问题空间说明。给每个 agent 分配不同的设计约束：

- Agent 1：“最小化接口——目标最多 1–3 个入口点。最大化每个入口点的杠杆。”
- Agent 2：“最大化灵活性——支持更多用例和扩展。”
- Agent 3：“围绕最常见的调用方优化——让默认路径极其简单。”
- Agent 4（如果适用）：“围绕跨接缝依赖的端口与适配器来设计。”

在简报中同时包含 [SKILL.md](SKILL.md) 词汇和 CONTEXT.md 词汇，这样每个 sub-agent 都能用项目的架构语言和领域语言一致地命名事物。

每个 sub-agent 输出：

1. 接口（类型、方法、参数——以及不变量、调用顺序、错误模式）
2. 展示调用方如何使用它的用法示例
3. 接缝背后隐藏了哪些实现
4. 依赖策略和适配器（见 [DEEPENING.md](DEEPENING.md)）
5. 权衡——哪里杠杆高，哪里杠杆薄

### 3. 展示和比较

按顺序展示各个设计，让用户能逐个消化，然后用文字进行比较。比较维度包括**深度**（接口上的杠杆）、**局部性**（变化集中在哪里）和**接缝位置**。

比较之后，给出你自己的建议：你认为哪个设计最强，以及原因。如果不同设计中的元素适合组合，就提出混合方案。要有明确立场——用户需要的是有判断力的建议，不是一份菜单。
