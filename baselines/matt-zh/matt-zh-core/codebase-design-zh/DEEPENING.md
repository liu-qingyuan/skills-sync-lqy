# 深化

在了解依赖关系的前提下，如何安全地深化一组浅模块。本文沿用 [SKILL.md](SKILL.md) 中的词汇 — **模块**、**接口**、**接缝**、**适配器**。

## 依赖类别

评估某个深化候选项时，先给它的依赖分类。依赖类别决定了深化后的模块如何跨接缝测试。

### 1. 进程内

纯计算、内存状态、没有 I/O。总是可以深化——把这些模块合并起来，然后直接通过新的接口测试。不需要适配器。

### 2. 本地可替代

有本地测试替身的依赖（比如用于 Postgres 的 PGLite、内存文件系统）。只要替身存在，就可以深化。深化后的模块在测试套件中使用这个替身运行测试。接缝是内部的；模块的外部接口不需要端口。

### 3. 远程但自有（端口与适配器）

你自己拥有、但跨网络边界调用的服务（微服务、内部 API）。在接缝处定义一个**端口**（接口）。深模块拥有业务逻辑；传输层作为**适配器**注入。测试使用内存适配器。生产环境使用 HTTP/gRPC/队列适配器。

建议表述：*“在接缝处定义端口，为生产实现 HTTP 适配器，为测试实现内存适配器。这样即使部署跨越网络，逻辑仍然位于一个深模块中。”*

### 4. 真正的外部依赖（Mock）

你无法控制的第三方服务（Stripe、Twilio 等）。深化后的模块把外部依赖作为注入端口接收；测试提供 mock 适配器。

## 接缝纪律

- **一个适配器意味着接缝还只是设想；两个适配器才说明接缝真实存在。** 除非至少有两个适配器有正当理由（通常是生产 + 测试），否则不要引入端口。只有一个适配器的接缝只是多一层间接。
- **区分内部接缝和外部接缝。** 深模块既可以有内部接缝（实现私有、供自身测试使用），也可以在接口处有外部接缝。不要仅仅因为测试会用到内部接缝，就把它们暴露到接口里。

## 测试策略：替换，不叠加

- 一旦深模块接口层面的测试已经存在，原来针对浅模块的单元测试就成了负担——删掉它们。
- 在深化后的模块接口上编写新测试。**接口就是测试表面**。
- 测试应通过接口断言可观察结果，而不是内部状态。
- 测试应能经受内部重构——它们描述的是行为，不是实现。如果实现一变测试就必须跟着变，说明测试越过了接口。
