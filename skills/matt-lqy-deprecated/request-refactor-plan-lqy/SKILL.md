当用户想要创建重构请求时，将调用此技能。您应该执行以下步骤。如果您认为没有必要，可以跳过这些步骤。

1. 请用户详细描述他们想解决的问题，以及他们已经想到的任何潜在方案。

2. 探索仓库以验证其断言并了解代码库的当前状态。

3. 询问用户是否考虑过其他方案，并主动提出可选方案。

4. 就实施情况追问用户。要非常详细和彻底。

5. 确定具体的实施范围。弄清楚你计划改变什么以及你计划不改变什么。

6. 查看代码库以检查代码库此区域的测试覆盖率。如果测试覆盖率不足，请询问用户他们的测试计划是什么。

7. 将实施分解为微小的提交计划。请记住 Martin Fowler 的建议：“使每个重构步骤尽可能小，以便您始终可以看到程序正在运行。”

8. 使用重构计划创建 GitHub 问题。使用以下模板进行问题描述：

<重构计划模板>

## 问题陈述

从开发商的角度来看，开发商面临的问题。

## 解决方案

问题的解决方案，从开发者的角度来看。

## 提交

一个长期、详细的实施计划。用简单中文编写计划，将实施分解为尽可能小的提交。每次提交都应该使代码库处于工作状态。

## 决定文件

已做出的实施决定的列表。这可以包括：

- 将要构建/修改的模块
- 需要修改的模块的接口
- 开发商的技术说明
- 架构决策
- 架构变更
- API合约
- 具体互动

不要包含特定的文件路径或代码片段。它们可能很快就会过时。

## 测试决策

所做的测试决定的列表。包括：

- 对什么是好的测试的描述（仅测试外部行为，而不测试实现细节）
- 将测试哪些模块
- 测试的现有技术（即代码库中类似类型的测试）

## 超出范围

对超出此重构范围的事物的描述。

## 进一步说明（可选）

有关重构的任何进一步说明。
