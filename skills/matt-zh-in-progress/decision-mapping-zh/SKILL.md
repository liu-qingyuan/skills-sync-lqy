# decision-mapping-zh

> 这是 Matt Pocock `decision-mapping` skill 的中文本地化版本。官方英文上游保留在 `skills/in-progress/decision-mapping`；本目录可按中文团队习惯继续调整。

## 本地化说明

- 优先用中文与用户沟通。
- 保留上游流程、检查点和文件约定。
- 如果本文件与上游英文版本冲突，以本中文版本为准；同步上游时先比较差异，再合并。

当一个松散的想法需要多个座席会话才能转化为计划时，就会调用此技能。它在 Markdown 文件中创建一个有状态的决策图，并驱动用户通过一系列票证来解决开放问题 - 这可能需要原型设计、研究或讨论。

## 决策图

决策图是一个紧凑的 Markdown 文件，每个规划工作一个，与项目一起进行 git 跟踪。它是规范的工件 - **整个地图作为上下文加载到每个会话中**，因此它必须保持紧凑。

在门票期间创建的资产应链接到地图，而不是在其中重复。

### 结构

编号条目（“门票”），每个条目都有自己的部分，并由其编号键入：
```markdown
## #1: Relational Or Non-Relational Database?

Blocked by: #<ticket-number>, #<ticket-number>
Type: Research | Prototype | Grilling

### Question

<question-here>

### Answer

<answer-here>
```
每张票证的大小必须适合一个 100K 令牌代理会话。

## 门票类型

门票分为三种类型：

- **研究**：阅读文档、第三方 API 或知识库等本地资源。创建 Markdown 摘要作为资产。当需要当前工作目录之外的知识时使用此选项。
- **原型**：编写 UI 或逻辑代码来测试假设或探索设计空间。使用 /prototype-zh 技能。创建原型作为资产。当“它应该看起来如何”或“它应该如何表现”是关键问题时使用此选项。
- **烧烤**：与特工对话。使用/烧烤和/域建模技能。一次问一个问题。默认情况。

## 战争迷雾

地图在边界之外是_故意_不完整的。你的工作是调查边境，并解决问题以推动边境向前发展。一次一个节点地拨开战争迷雾。

在某个时刻，战争的迷雾应该已经被推得足够远，以至于通往终点线的道路已经清晰可见。到那时，将不需要更多的票证，并且决策图可以被视为“完成”。

## 调用

有两种方法可以调用此技能：**引导**和**恢复**。

### 引导程序

用户以松散的想法调用。

1. 运行 /grilling-zh + /domain-modeling-zh 会话来公开未决决策。一次问一个问题。
2. 编写一个新的决策图——主要是雾、边界确定、内联解析的微不足道的可判定条目。
3. 停止。地图构建是一次会议的工作；也不解决票证。

### 简历

用户使用现有地图的路径和票号进行调用。

1. 加载**整个地图**作为上下文。
2. 运行会话来解决故障单，根据需要调用技能。如果有疑问，请使用“/griling”和“/domain-modeling-zh”。
3. 在票证正文中记录会话解决的问题。
4. 添加新发现的票证（具有正确的“blocked_by”边缘）。
5. 停下来。

如果做出的决策使地图的其他部分无效，请更新或删除这些节点。

## 并行性

用户可以选择并行运行票证，因此预计其他代理会对地图进行更改。

## 跳过决策图

很多时候，最初的拷问不会导致战争迷雾。没有未解决的票证。除了执行之外，无事可做。

在这些情况下，您应该为用户提供跳过决策图的机会 - 因为仅当需要做出多会话决策时才需要决策图。

如果他们跳过它，您应该建议直接实施或使用“/to-prd-zh”来安排多会话实施。
