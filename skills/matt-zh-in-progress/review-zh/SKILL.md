对“HEAD”和用户提供的固定点之间的差异进行两轴审查：

- **标准** — 代码是否符合此仓库记录的编码标准？
- **规范** — 代码是否忠实地实现了原始问题/PRD/规范？

两个轴都作为**并行sub-agent**运行，因此它们不会污染彼此的上下文，然后此技能会聚合他们的发现。

issue tracker应该已提供给您 - 如果缺少“docs/agents/issue-tracker.md”，请运行“/setup-matt-pocock-skills-zh”。

## 流程

### 1. 固定固定点

用户所说的都是固定点——提交 SHA、分支名称、标签、“main”、“HEAD~5”等。如果他们没有指定，请询问。

捕获一次 diff 命令：`git diff <fixed-point>...HEAD`（三点，因此比较是针对合并基础的）。另请注意通过“git log <fixed-point>..HEAD --oneline”提交的列表。

在进一步之前，确认定点解析（`git rev-parse <fixed-point>`）并且 diff 不为空。错误的引用或空的差异应该在这里失败——而不是在两个并行的sub-agent内部。

### 2. 确定规范来源

按以下顺序查找原始规范：

1. 提交消息中的问题引用（`#123`、`Closes #45`、GitLab `!67` 等）- 通过 `docs/agents/issue-tracker.md` 中的工作流获取。
2. 用户作为参数传递的路径。
3. `docs/`、`specs/` 或 `.scratch/` 下与分支名称或功能匹配的 PRD/spec 文件。
4. 如果没有找到任何内容，请询问用户规格在哪里。如果他们说没有，**规格**sub-agent将跳过并报告“没有可用的规格”。

### 3. 确定标准来源

仓库中记录如何编写代码的任何内容，例如“CODING_STANDARDS.md”或“CONTRIBUTING.md”。

### 4. 并行生成两个sub-agent

通过两次“代理”工具调用发送一条消息。两者都使用“通用”sub-agent。

**标准sub-agent提示** — 包括：

- 完整的 diff 命令和提交列表。
- 您在步骤 3 中找到的标准源文件列表。
- 简介：“报告 - 每个文件/相关块 - 差异违反记录标准的每个地方。引用标准（文件 + 规则）。区分严重违规和判断调用。跳过工具强制执行的任何内容。少于 400 字。”

**规格sub-agent提示** — 包括：

- diff 命令和提交列表。
- 规范的路径或获取的内容。
- 简介：“报告：(a) 规范要求的要求缺失或不完整；(b) 差异中未要求的行为（范围蔓延）；(c) 看似已实现但实现看起来错误的要求。引用每个发现的规范行。少于 400 字。”

如果缺少规格，请跳过规格sub-agent并在最终报告中注明这一点。

### 5. 聚合

在“## 标准”和“## 规范”标题下逐字或稍加修改地呈现两份报告。 **不要**合并或重新排列结果——两个轴是故意分开的（请参阅_为什么是两个轴_）。

以一行总结结束：每个轴的总发现结果，以及每个轴内最严重的问题（如果有）。不要跨轴选出一个获胜者——这是分离所要防止的重新排名。

## 为什么有两个轴

更改可以通过一个轴而使另一个轴失败：

- 遵循每个标准但实现了错误的代码 → **标准通过，规范失败。**
- 代码完全按照问题要求执行，但违反了项目约定 → **规范通过，标准失败。**

单独报告它们可以防止一个轴掩盖另一个轴。
