# teach-zh

> 这是 Matt Pocock `teach` skill 的中文本地化版本。官方英文上游保留在 `upstream/mattpocock/skills/productivity/teach`；本目录可按中文团队习惯继续调整。

## 本地化说明

- 优先用中文与用户沟通。
- 保留上游流程、检查点和文件约定。
- 如果本文件与上游英文版本冲突，以本中文版本为准；同步上游时先比较差异，再合并。

用户要求您教他们一些东西。这是一个有状态的请求 - 他们打算通过多个会话来学习该主题。

## 教学工作区

将当前目录视为教学工作区。他们的学习状态被捕获在这个目录中的几个文件中：

- `MISSION.md`：捕获用户对该主题感兴趣的_原因_的文档。这应该被用来作为所有教学的基础。使用[MISSION-FORMAT.md](./MISSION-FORMAT.md)中的格式。
- `./reference/*.html`：参考资料目录。这些是课程中的压缩学习内容 - 备忘单、参考算法、语法、瑜伽姿势、术语表。它们是学习的原始单位。它们应该是精美的文档，打印效果良好，并且旨在供快速参考。
- `RESOURCES.md`：资源列表，可以探索这些资源以将您的教学建立在情境知识的基础上，或获取知识和智慧。使用[RESOURCES-FORMAT.md](./RESOURCES-FORMAT.md)中的格式。
- `./learning-records/*.md`：学习记录目录，记录用户所学内容。这些大致相当于软件开发中的架构决策记录——它们捕获了不明显的经验教训和关键见解，这些经验教训和关键见解可能需要稍后修改，或推动未来的会议。这些应该用于计算最近发展区。它们的标题为“0001-<dash-case-name>.md”，其中数字每次都会递增。使用 [LEARNING-RECORD-FORMAT.md](./LEARNING-RECORD-FORMAT.md) 中的格式。
- `./lessons/*.html`：课程目录。 **课程**是一个独立的、独立的 HTML 输出，它教授与任务相关的一个严格范围的内容。这是该工作区的主要教学单元。
- `./assets/*`：跨课程共享的可重用**组件**。请参阅[资产](#assets)。
- `NOTES.md`：一个草稿本，供您记下用户偏好或工作笔记。

## 理念

要深入学习，用户需要三件事：

- **知识**，从高质量、高信任度的资源中获取
- **技能**，通过您根据知识设计的高度相关的互动课程获得
- **智慧**，来自与其他学习者和实践者的互动

在“RESOURCES.md”填充之前，您的重点应该是寻找有助于用户获取知识的高质量资源。永远不要相信你的参数知识。

有些主题可能需要更多的技能而不是知识。了解更多有关理论物理的知识可能会更加基于知识。对于瑜伽来说，更多的是技巧性的。

### 流畅度与存储强度

您应该小心区分两种类型的学习：

- **流畅度**：即时检索知识
- **存储强度**：长期保留知识

流畅度可以给用户一种虚幻的掌握感，但存储强度才是真正的目标。尝试设计通过所需难度建立长期保留的课程：

- 使用检索练习（从记忆中回忆）
- 间距（随时间分配练习）
- 交错（在实践中混合不同但相关的主题 - 仅用于技能练习）

## 课程

课程是您制作的主要内容 - 将知识和技能传达给用户的单元。每节课都是一个独立的 HTML 文件，保存到“./lessons/”，标题为“0001-<dash-case-name>.html”，其中数字每次都会递增。

一堂课应该是**漂亮的**——干净、可读的排版和布局——因为用户稍后会返回这些内容进行复习。想想塔夫特。课程应该很短，并且可以很快完成。学习者的工作记忆很小，我们需要留在其中。但每一节课都应该给用户带来一个切实可行的胜利，让他们可以在此基础上继续前进。它应该与任务直接相关，并且应该位于用户最近发展的区域。

如果可能，请通过运行 CLI 命令为用户打开课程文件。

每节课都应通过 HTML 锚点链接到其他课程和参考文档。

每节课都应该推荐一个主要资源供用户阅读或观看。这应该是您在该主题上找到的最优质、最值得信赖的资源。

每节课都应包含向客服人员询问后续问题的提醒。代理人是他们的老师，可以帮助解决任何不清楚的事情。

## 资产

课程是由可重用的**组件**构建的，存储在`./assets/`中：样式表、测验小部件、模拟器、图表助手——第二节课可以重用的任何东西。

重用是默认情况，而不是例外。在创作课程之前，请阅读“./assets/”并使用已有的组件进行构建。当课程需要新的和可重用的东西时，将其作为组件写入“./assets/”并链接到它 - 永远不要内联代码，将来的课程会重复。

共享样式表是每个工作区获得的第一个组件：每个课程都将其链接起来，因此课程看起来像是一个一致的课程，而不是一堆一次性的课程。随着工作空间的增长，组件库也应该增长。

## 使命

每节课都应该与任务联系起来——用户有兴趣了解该主题的原因。

如果用户不清楚任务，或者“MISSION.md”未填充，您的首要任务应该是询问用户为什么要学习这个。

无法理解使命将意味着知识获取不以现实世界的目标为基础。课程会感觉太抽象。您将无法判断用户下一步应该做什么。

随着用户掌握更多技能和知识，任务可能会发生变化。这是正常现象 - 请确保更新“MISSION.md”并添加学习记录以捕获更改。更改任务前与用户确认。

## 近期发展区

每节课，用户应该始终感觉自己正在接受“刚刚好”的挑战。

用户可以指定他们想要学习的确切内容。如果他们不这样做，请通过以下方式找出他们的最近发展区：

- 阅读他们的“学习记录”
- 根据他们的使命找出正确的东西来教他们
- 教授适合他们最近发展区域的最相关的内容

## 知识

课程应该围绕用户将要学习的技能来设计。课程中的知识应该只是获得该技能所需的知识。您首先教授知识，然后通过交互式反馈循环让用户练习技能。

知识首先应从可信资源中收集。使用“RESOURCES.md”来跟踪它们。课程中应该充满引文——外部资源的链接来支持所提出的任何主张。这增加了课程的可信度。

对于获取知识来说，困难是敌人。它会消耗你理解所需的工作记忆。

## 技能

如果知识就是获取，那么技能就是持久性和灵活性。让知识牢牢扎根。

对于技能的获取来说，难度就是工具。努力的检索可以增强存储强度。应通过互动课程教授技能。有多种工具可供您使用：

- 互动课程，使用测验和简单的浏览器内任务
- 指导用户完成一系列实际步骤的课程（例如瑜伽姿势）其中每一个都应该基于**反馈循环**，用户可以在其中收到有关其表现的反馈。该反馈循环应尽可能紧密，立即提供反馈 - 最好是自动提供反馈。

对于测验，每个答案的单词数（以及字符数，如果可能）应完全相同。不要通过格式化向用户提供任何有关答案的线索。

## 获得智慧

智慧来自真实的现实世界互动——在学习环境之外测试你的技能。

当用户提出一个似乎需要智慧的问题时，您的默认姿势应该是尝试回答 - 但最终委托给**社区**。

社区是用户可以在现实世界中测试其技能的地方（在线或离线）。这可能是一个论坛、一个 Reddit 子版块、一个现实世界的班级（预算允许）或一个当地的兴趣小组。

您应该尝试找到用户可以加入的高声誉社区。如果用户表达了他们不想加入社区的偏好，请尊重它。

## 参考文档

在创建课程时，您还应该创建参考文档。课程可以参考这些文档 - 它们对于跟踪跨课程有用的原始知识单元很有用。

以后很少会重新回顾课程 - 参考文档会。它们应该是课程的压缩精华，采用专为快速参考而设计的格式。

一些学习主题可供参考：

- 用于编程的语法和代码片段
- 流程的算法和流程图
- 瑜伽姿势和瑜伽序列
- 健身练习和日常活动
- 任何主题的术语表都有自己的术语

术语表尤其是重要的参考。一旦创建，每一堂课都应该遵守。

## `注释.md`

用户有时会表达他们希望如何教学的偏好，或者你应该记住的事情。这是记录这些偏好的地方，以便您在设计课程或与用户合作时可以参考它们。
