# MISSION.md 格式

`MISSION.md` 位于工作空间根目录。它捕获了用户学习该主题的_原因_。每个教学决策——接下来教什么、展示哪些资源、设计哪些练习——都应该追溯到这个文件。

＃＃ 模板
```md
# Mission: {Topic}

## Why
{1-3 sentences. The concrete real-world goal the user is chasing. What changes in their life or work when they have this skill? Avoid abstract framings like "to understand X" — push for the underlying outcome.}

## Success looks like
- {A specific, observable thing the user will be able to do}
- {Another specific thing}
- {…}

## Constraints
- {Time, budget, prior commitments, learning preferences, anything that bounds the approach}

## Out of scope
- {Adjacent topics the user explicitly does not want to chase right now — protects the zone of proximal development}
```
## 规则

- **每个工作区一个任务。** 如果用户想要学习两个不相关的事物，那就是两个工作区。
- **具体胜于抽象。** “十月之前跑一次半程马拉松”胜过“变得更健康”。 “向我的团队提供 Rust CLI”胜过“学习 Rust”。
- **拒绝含糊不清。** 如果用户无法阐明原因，请在编写任何内容之前追问他们。糟糕的使命比没有使命更糟糕。
- **当现实发生变化时进行修改。** 任务发生变化。当用户的目标发生变化时，更新此文件 - 不要留下陈旧的任务来引导未来的会话。
- **保持简短。** 如果“MISSION.md”跑过屏幕，它就不再是一个指南针，而是开始成为一个计划。
