# 分类

通过分类角色的小型状态机将问题转移到项目问题跟踪器上。

如果此存储库将外部拉取请求视为请求表面（请参阅问题跟踪器配置），分类也会涵盖它们：**PR 是带有附加代码的问题** - 相同的角色，相同的状态，相同的机器，下面有一些标记为“用于 PR”的增量。根据跟踪器配置解决问题或 PR 的裸“#42”。

在分类期间发布到问题跟踪器的每个评论或问题**必须**以此免责声明开头：
```
> *This was generated by AI during triage.*
```

## 角色

两个**类别**角色：

- `bug` — 某些东西坏了
- `enhancement` — 新功能或改进

五个**状态**角色：

- `needs-triage` — 维护者需要评估
- `needs-info` — 等待记者了解更多信息
- `ready-for-agent` — 完全指定，准备好 AFK 特工
- “为人类做好准备”——需要人类实施
- `wontfix` — 不会被执行

对于 PR，针对所附代码读取相同的状态：“准备好代理”意味着已附加摘要，代理应根据差异采取下一步； “为人类做好准备”意味着它已经准备好与人类融合。

每个分类的问题都应该恰好具有一种类别角色和一种国家角色。如果状态角色发生冲突，请标记它并在执行其他操作之前询问维护人员。

这些是规范的角色名称 - 问题跟踪器中使用的实际标签字符串可能有所不同。应该已经向您提供了映射 - 如果没有，请运行“/setup-matt-pocock-skills-zh”。

状态转换：未标记的问题通常首先进入“需求分类”；从那里它移动到“需求信息”、“准备代理”、“准备人员”或“不会修复”。一旦记者回复，“需求信息”就会返回“需求分类”。维护者可以随时覆盖 - 标记看起来不寻常的转换并在继续之前询问。

## 调用

维护者调用“/triage-zh”并用自然语言描述他们想要什么。解释请求并采取行动。示例：

- “告诉我任何需要我注意的事情”
- “让我们看看#42”（问题或 PR）
- “将#42移至准备好代理处”
- “有什么准备好让特工来接货吗？”

## 显示需要注意的地方

查询问题跟踪器并显示三个存储桶，最旧的在前：

1. **未标记** — 从未进行分类。
2. **`需求分类`** — 评估正在进行中。
3. **自上次分类记录以来记者活动的“需求信息”** - 需要重新评估。

当 PR 在范围内时，将外部 PR 包含在这些存储桶中，并标记每行“[PR]”或“[issue]”。 Discovery 仅显示*外部* PR（跟踪器配置定义了谁算作外部）——协作者的正在进行的 PR 不是分类工作。此过滤器仅用于发现；无论作者是谁，都会对明确命名的 PR 进行分类。

显示每个项目的计数和一行摘要。让维护者选择。

## 对特定问题或 PR 进行分类

1. **收集上下文。** 阅读完整的问题或 PR（正文、评论、标签、作者、日期；对于 PR，还包括差异）。解析之前的任何分类记录，这样您就不会再次提出已解决的问题。使用项目的领域术语表探索代码库，尊重该领域的 ADR。针对代码库运行两项检查： (a) **冗余** — 按领域概念（不仅仅是请求的措辞）搜索所请求行为的现有实现，并报告您查看的位置。如果找到，则它是一个已经实现的“wontfix”（步骤 5）。 (b) **事先拒绝** — 阅读“.out-of-scope/*.md”并显示任何与此请求类似的内容。

2. **推荐。** 告诉维护人员您的类别并说明推荐的理由，以及与请求相关的简短代码库摘要 - 包括它是否已经实现。等待方向。

3. **验证该声明。** 在进行任何烧烤之前，请检查该声明是否成立。对于错误，请按照报告者的步骤重现它。对于 PR，确认 diff 执行其声称的操作 - 检查它，运行相关测试或命令。报告发生的情况：已确认（使用代码路径）、失败或细节不足（强烈的“需求信息”信号）。经确认的验证使代理简介更加有力。4. **烧烤（如果需要）。** 如果请求需要充实，请同时运行“/烧烤”和“/领域建模”技能 - 一次将其烧烤成一个问题，锐化领域术语并在决策落地时更新内嵌的“CONTEXT.md”/ADR。

5. **应用结果：**
   - `准备好代理` — 发布代理简短评论 ([AGENT-BRIEF.md](AGENT-BRIEF.md))。
   -“为人类做好准备”——与代理简介相同的结构，但请注意为什么它不能被委托（判断调用、外部访问、设计决策、手动测试）。
   - `needs-info` — 发布分类注释（模板如下）。
   - `wontfix` — 关闭，注释取决于*原因*：
     - **已实施** — 更改已存在于代码库中。指向它居住的地方； **不要**写入`.out-of-scope/`（该知识库用于*拒绝的*请求，而不是构建的请求）。
     - **拒绝（bug）** - 礼貌的解释，然后关闭。
     - **拒绝（增强）** — 写入 `.out-of-scope/`，从评论链接到它，然后关闭 ([OUT-OF-SCOPE.md](OUT-OF-SCOPE.md))。
   - `needs-triage` — 应用角色。如果有部分进展，可选择发表评论。

## 快速状态覆盖

如果维护者说“将 #42 移至准备代理”，请信任他们并直接应用该角色。确认您要做什么（角色变更、评论、关闭），然后采取行动。跳过烧烤。如果在没有盘问会议的情况下进入“准备代理”，询问他们是否想写一份代理简介。

## 需求信息模板
```markdown
## Triage Notes

**What we've established so far:**

- point 1
- point 2

**What we still need from you (@reporter):**

- question 1
- question 2
```
将烧烤期间解决的所有问题记录在“到目前为止已建立”下，这样工作就不会丢失。问题必须具体且可操作，而不是“请提供更多信息”。

## 恢复上一个会话

如果问题或 PR 存在先前的分类注释，请阅读它们，检查记者是否回答了任何悬而未决的问题，并在继续之前提供更新的图片。不要再次提出已解决的问题。
