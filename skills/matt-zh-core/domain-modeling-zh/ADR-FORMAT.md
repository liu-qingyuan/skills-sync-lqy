# ADR 格式

ADR 位于“docs/adr/”中并使用顺序编号：“0001-slug.md”、“0002-slug.md”等。

仅当需要第一个 ADR 时才懒惰地创建“docs/adr/”目录。

＃＃ 模板
```md
# {Short title of the decision}

{1-3 sentences: what's the context, what did we decide, and why.}
```
就是这样。 ADR 可以是单个段落。价值在于记录*做出的决定以及*为什么*——而不是填写部分。

## 可选部分

仅当它们能增加真正的价值时才包含它们。大多数 ADR 不需要它们。

- **状态** frontmatter（“提议|接受|不推荐|被ADR-NNNN取代”）——在重新审视决策时有用
- **考虑的选项** - 仅当被拒绝的替代方案值得记住时
- **后果** - 仅当需要消除不明显的下游影响时

## 编号

扫描“docs/adr/”以查找现有的最高数字并加一。

## 何时提供 ADR

所有这三项都必须为真：

1. **难以逆转**——以后改变主意的成本是有意义的
2. **没有上下文就令人惊讶** - 未来的读者在查看代码时会想知道“他们到底为什么这样做？”
3. **真正权衡的结果** - 有真正的替代方案，并且您出于特定原因选择了一个

如果一个决定很容易推翻，那就跳过它——你只会推翻它。如果这并不奇怪，没有人会想知道为什么。如果没有真正的替代方案，除了“我们做了显而易见的事情”之外，就没有什么可记录的了。

### 符合什么资格

- **建筑形状。**“我们正在使用 monorepo。” “写入模型是事件源的，读取模型被投影到 Postgres 中。”
- **上下文之间的集成模式。**“订购和计费通过域事件进行通信，而不是同步 HTTP。”
- **带有锁定的技术选择。**数据库、消息总线、身份验证提供者、部署目标。不是每个图书馆——只是那些需要四分之一的时间才能更换的图书馆。
- **边界和范围决策。**“客户数据由客户上下文拥有；其他上下文仅通过 ID 引用它。”明确的“不”与“是”一样有价值。
- **故意偏离明显的路径。** “我们使用手动 SQL 而不是 ORM，因为 X。”任何理性的读者都会做出相反的假设。这些会阻止下一位工程师“修复”故意的东西。
- **代码中不可见的约束。**“由于合规性要求，我们无法使用 AWS。” “根据合作伙伴 API 合同，响应时间必须低于 200 毫秒。”
- **当拒绝不明显时拒绝替代方案。** 如果您出于微妙的原因考虑过 GraphQL 并选择了 REST，请记录下来 - 否则有人会在六个月内再次建议 GraphQL。
