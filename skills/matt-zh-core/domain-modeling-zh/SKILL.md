# 领域建模

在设计时积极构建和完善项目的领域模型。这是“积极的”学科——挑战术语，发明边缘场景，并在术语表和决策具体化时将其写下来。 （仅仅*阅读*`CONTEXT.md`来获取词汇并不是这项技能——这是任何技能都可以做到的单行习惯。这项技能适用于当你改变模型时，而不仅仅是使用它。）

## 文件结构

大多数仓库都有一个上下文：
```
/
├── CONTEXT.md
├── docs/
│   └── adr/
│       ├── 0001-event-sourced-orders.md
│       └── 0002-postgres-for-write-model.md
└── src/
```
如果根目录存在“CONTEXT-MAP.md”，则该仓库具有多个上下文。地图指向每个人居住的地方：
```
/
├── CONTEXT-MAP.md
├── docs/
│   └── adr/                          ← system-wide decisions
├── src/
│   ├── ordering/
│   │   ├── CONTEXT.md
│   │   └── docs/adr/                 ← context-specific decisions
│   └── billing/
│       ├── CONTEXT.md
│       └── docs/adr/
```
懒惰地创建文件——仅当您有东西要写时。如果不存在“CONTEXT.md”，请在第一个术语解析后创建一个。如果不存在“docs/adr/”，请在需要第一个 ADR 时创建它。

## 会议期间

### 挑战词汇表

当用户使用与 CONTEXT.md 中现有语言冲突的术语时，请立即指出。 “你的词汇表将‘取消’定义为 X，但你的意思似乎是 Y——是哪一个？”

### 锐化模糊语言

当用户使用模糊或过多的术语时，建议一个精确的规范术语。 “你说的‘账户’——你指的是客户还是用户？它们是不同的东西。”

### 讨论具体场景

当讨论域关系时，用特定的场景对它们进行压力测试。发明一些场景来探测边缘情况并迫使用户精确地了解概念之间的界限。

### 与代码的交叉引用

当用户陈述某事如何工作时，检查代码是否一致。如果您发现矛盾，请指出：“您的代码取消了整个订单，但您刚刚说可以部分取消 - 这是对的吗？”

### 内联更新 CONTEXT.md

解决术语后，立即更新“CONTEXT.md”。不要将它们分批处理——在它们发生时捕获它们。使用[CONTEXT-FORMAT.md](./CONTEXT-FORMAT.md)中的格式。

`CONTEXT.md` 应该完全没有实现细节。不要将“CONTEXT.md”视为规范、草稿本或实施决策的仓库。它只是一个术语表，仅此而已。

### 谨慎提供 ADR

仅当这三个条件都满足时才会提出创建 ADR：

1. **难以逆转**——以后改变主意的成本是有意义的
2. **没有上下文就令人惊讶** - 未来的读者会想知道“他们为什么这样做？”
3. **真正权衡的结果** - 有真正的替代方案，并且您出于特定原因选择了一个

如果缺少三者中的任何一个，请跳过 ADR。使用[ADR-FORMAT.md](./ADR-FORMAT.md)中的格式。
