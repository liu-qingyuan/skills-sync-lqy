# 代码库设计

设计**深层模块**：小界面背后的许多行为，放置在干净的缝隙中，可通过该界面进行测试。无论何时设计或重构代码，都请使用这种语言和这些原则。目的是为调用者提供杠杆作用，为维护者提供本地化，并为每个人提供可测试性。

## 术语表

准确使用这些术语 — 不要替换“组件”、“服务”、“API”或“边界”。一致的语言才是重点。

**模块** — 任何具有接口和实现的东西。故意与规模无关：函数、类、包或跨层切片。 _避免_：单元、组件、服务。

**接口** - 调用者必须知道才能正确使用模块的所有内容：类型签名，还有不变量、排序约束、错误模式、所需的配置和性能特征。 _避免_：API、签名（太窄——它们仅指类型级别的表面）。

**实现** — 模块内部的内容，其代码主体。与 **Adapter** 不同：事物可以是具有大型实现的小型适配器（Postgres 存储库），也可以是具有小型实现的大型适配器（内存中的伪造）。当接缝成为主题时，请使用“适配器”；否则“实施”。

**深度** - 接口的影响力：调用者（或测试）可以在他们必须学习的每个接口单元中执行的行为量。当大量行为位于一个小接口后面时，模块是**深**；当接口几乎与实现一样复杂时，模块是**浅**。

**Seam** _(Michael Feathers)_ — 您无需在该位置进行编辑即可更改行为的地方；模块接口所在的*位置*。接缝放置在哪里是它自己的设计决定，与其背后的内容不同。 _避免_：边界（重载了 DDD 的有界上下文）。

**适配器** — 满足接口要求的具体事物。描述*角色*（它填充什么位置），而不是内容（里面有什么）。

**杠杆**——调用者从深度获得的东西：他们学习的每个接口单元的更多功能。一项实施可在 N 个调用站点和 M 个测试中获得回报。

**局部性** - 维护人员从深度获得的信息：更改、错误、知识和验证集中在一个地方，而不是分散在调用者之间。修复一次，到处修复。

## 深与浅

**深层模块** = 小接口 + 大量实现：
```
┌─────────────────────┐
│   Small Interface   │  ← Few methods, simple params
├─────────────────────┤
│                     │
│  Deep Implementation│  ← Complex logic hidden
│                     │
└─────────────────────┘
```
**浅模块**=大接口+小实现（避免）：
```
┌─────────────────────────────────┐
│       Large Interface           │  ← Many methods, complex params
├─────────────────────────────────┤
│  Thin Implementation            │  ← Just passes through
└─────────────────────────────────┘
```
设计界面时，问：

- 我可以减少方法的数量吗？
- 我可以简化参数吗？
- 我可以在里面隐藏更多的复杂性吗？

## 原则

- **深度是接口的属性，而不是实现。** 深度模块可以在内部由小的、可模拟的、可交换的部分组成——它们只是不是接口的一部分。模块可以具有**内部接缝**（其实现私有，由其自己的测试使用）以及其接口处的**外部接缝**。
- **删除测试。**想象一下删除模块。如果复杂性消失了，那就是一个传递。如果复杂性在 N 个调用者中再次出现，那么它就值得了。
- **接口是测试表面。** 调用者和测试跨越同一接缝。如果您想测试*过去*的接口，则模块的形状可能是错误的。
- **一个适配器意味着一条假设的接缝。两个适配器意味着一个真正的适配器。** 不要引入接缝，除非接缝上确实有变化。

## 可测试性设计

良好的界面使测试变得自然：

1. **接受依赖项，而不是创建它们。**
```typescript
   // Testable
   function processOrder(order, paymentGateway) {}

   // Hard to test
   function processOrder(order) {
     const gateway = new StripeGateway();
   }
   ```
2. **返回结果，不产生副作用。**
```typescript
   // Testable
   function calculateDiscount(cart): Discount {}

   // Hard to test
   function applyDiscount(cart): void {
     cart.total -= discount;
   }
   ```
3. **表面积小。** 更少的方法 = 需要更少的测试。更少的参数 = 更简单的测试设置。

## 关系

- **模块**只有一个**接口**（它呈现给调用者和测试的表面）。
- **深度**是**模块**的属性，根据其**接口**进行测量。
- **接缝**是**模块**的**接口**所在的地方。
- **适配器**位于**接缝**并满足**接口**。
- **深度**为调用者产生**杠杆**，为维护者产生**局部性**。

## 拒绝的框架

- **深度为实现线与接口线的比率** (Ousterhout)：奖励填充实现。我们使用深度作为杠杆。
- **“Interface”作为 TypeScript `interface` 关键字或类的公共方法**：太窄 - 这里的接口包括调用者必须知道的所有事实。
- **“边界”**：用 DDD 的有界上下文重载。说**接缝**或**界面**。
