# 深化

考虑到其依赖性，如何安全地深化浅层模块集群。假定 [SKILL.md](SKILL.md) 中的词汇 — **模块**、**接口**、**接缝**、**适配器**。

## 依赖类别

在评估加深候选项时，对其依赖性进行分类。该类别决定了如何跨接缝测试加深的模块。

### 1. 进程中

纯计算，内存状态，无 I/O。始终可深化——合并模块并直接通过新界面进行测试。无需适配器。

### 2.本地可替代

具有本地测试替代品的依赖项（PGLite for Postgres，内存文件系统）。如果替代存在则可加深。加深的模块通过测试套件中运行的替代程序进行测试。接缝是内部的；模块外部接口无端口。

### 3. 远程但拥有（端口和适配器）

您自己的跨网络边界的服务（微服务、内部 API）。在接缝处定义**端口**（接口）。深层模块拥有逻辑；传输作为 **适配器** 注入。测试使用内存适配器。生产使用 HTTP/gRPC/队列适配器。

建议形式：*“在接缝处定义一个端口，实现一个用于生产的 HTTP 适配器和一个用于测试的内存适配器，因此逻辑位于一个深层模块中，即使它是跨网络部署的。”*

### 4.真正的外部（模拟）

您无法控制的第三方服务（Stripe、Twilio 等）。模块加深将外部依赖作为注入端口；测试提供了一个模拟适配器。

## 接缝纪律

- **一个适配器意味着一条假设的接缝。两个适配器意味着一个真正的适配器。** 不要引入端口，除非至少有两个适配器是合理的（通常是生产 + 测试）。单适配器接缝只是间接的。
- **内部接缝与外部接缝。** 深层模块可以具有内部接缝（其实现私有，由其自己的测试使用）以及其接口处的外部接缝。不要仅仅因为测试使用它们就通过接口暴露内部接缝。

## 测试策略：替换，不分层

- 一旦加深模块接口上的测试存在，浅层模块上的旧单元测试就会变得浪费——删除它们。
- 在模块加深的界面上编写新的测试。 **界面为测试面**。
- 测试通过接口断言可观察的结果，而不是内部状态。
- 测试应该在内部重构中幸存下来——它们描述的是行为，而不是实现。如果当实现改变时测试也必须改变，那么它就是超越接口的测试。
