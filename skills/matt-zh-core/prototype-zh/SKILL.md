# 原型

原型是**回答问题的一次性代码**。问题决定形状。

## 选择一个分支

确定正在回答哪个问题 - 根据用户的提示、周围的代码，或者通过询问用户是否在附近：

- **“这个逻辑/状态模型感觉正确吗？”** → [LOGIC.md](LOGIC.md)。构建一个小型交互式终端应用程序，推动状态机完成纸面上难以推理的情况。
- **“这应该是什么样子？”** → [UI.md](UI.md)。在单个路由上生成几个完全不同的 UI 变体，可通过 URL 搜索参数和浮动底部栏进行切换。

这两个分支产生非常不同的工件——犯这个错误会浪费整个原型。如果问题确实不明确并且用户无法访问，则默认选择与周围代码更匹配的分支（后端模块 → 逻辑；页面或组件 → UI），并在原型顶部陈述假设。

## 适用于两者的规则

1. **从第一天开始就扔掉，并明确标记为这样。** 将原型代码放置在靠近实际使用位置的地方（在其原型设计的模块或页面旁边），这样上下文就很明显 - 但命名它以便普通读者可以看到它是原型，而不是生产。对于一次性 UI 路由，请遵守项目已使用的任何路由约定；不要发明新的顶层结构。
2. **一个命令即可运行。** 无论项目的现有任务运行程序支持什么 - `pnpm <name>`、`python <path>`、`bun <path>` 等。用户必须能够不假思索地启动它。
3. **默认情况下没有持久性。** 状态存在于内存中。持久性是原型要_检查_的东西，而不是它应该依赖的东西。如果问题明确涉及数据库，请点击临时数据库或带有明确“PROTOTYPE —擦除我”名称的本地文件。
4. **跳过修饰。** 除了使原型_可运行_之外，没有测试，没有错误处理，没有抽象。重点是快速学习一些东西，然后删除它。
5. **显示状态。** 在每个操作（逻辑）或每个变体开关（UI）之后，打印或渲染完整的相关状态，以便用户可以看到发生了什么变化。
6. **完成后删除或吸收。** 当原型回答了它的问题时，要么删除它，要么将经过验证的决策折叠到真实代码中 - 不要让它在存储库中腐烂。

## 完成后

_answer_是原型中唯一值得保留的东西。将其捕获在持久的地方（提交消息、ADR、问题或原型旁边的“NOTES.md”）以及它正在回答的问题。如果用户在附近，那么捕获就是一个快速的对话；如果没有，请保留占位符，以便他们（或您，在下一次传递时）可以在删除原型之前填写结论。
