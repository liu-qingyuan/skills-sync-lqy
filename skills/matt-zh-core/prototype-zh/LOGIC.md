# 逻辑原型

一个微型交互式终端应用程序，可让用户手动驱动状态模型。当问题涉及**业务逻辑、状态转换或数据形状**时，请使用此选项——这种事情在纸面上看起来很合理，但只有在实际案例中推演时才会感觉错误。

## 当这是正确的形状时

- “我不确定这个状态机是否处理 X 然后 Y 的边缘情况。”
- “这个数据模型实际上让我代表......的情况吗？”
- “我想在编写 API 之前先了解一下 API 应该是什么样子。”
- 用户想要**按下按钮并观察状态变化**的任何内容。

如果问题是“这应该是什么样子”——错误的分支。使用 [UI.md](UI.md)。

## 流程

### 1. 陈述问题

在编写代码之前，写下您要原型设计的状态模型和问题。原型自述文件中的一段或文件顶部的注释。回答错误问题的逻辑原型纯粹是浪费——将问题明确化，以便以后可以检查，无论用户是现在观看还是AFK返回。

### 2.选择语言

使用宿主项目使用的任何内容。如果项目没有明显的运行时（例如文档仓库），请询问。

匹配项目现有的工具约定 - 不要仅为原型添加新的包管理器或运行时。

### 3. 隔离可移植模块中的逻辑

将实际的逻辑（回答问题的部分）放在一个小的、纯粹的界面后面，稍后可以将其取出并放入真正的代码库中。周围的 TUI 都是一次性的；逻辑模块不应该是。

正确的形状取决于问题：

- **一个纯粹的减速器** - `(state, action) => state`。当操作是离散事件并且状态是单个值时，效果很好。
- **状态机** — 显式状态和转换。当“哪些行为现在是合法的”成为问题的一部分时，这很好。
- **一小组基于纯数据类型的纯函数**。当没有隐式当前状态——只有转换时，这很好。
- **当逻辑真正拥有持续的内部状态时，具有清晰方法表面的类或模块**。

选择最适合所提出问题的形状，*而不是*最容易连接到 TUI 的形状。保持纯粹：没有 I/O，没有终端代码，没有用于控制流的“console.log”。 TUI 导入它并调用它；没有任何东西流向另一个方向。

这就是原型在其生命周期后仍然有用的原因。当问题得到解答后，经过验证的减速器/机器/功能集可以提升到真实模块中 - TUI shell 被删除。

### 4. 构建公开状态的最小 TUI

将其构建为 **轻量级 TUI** — 在每次勾选时，清除屏幕（`console.clear()` / `print("\033[2J\033[H")` / 等效项）并重新渲染整个框架。用户应该始终看到一个稳定的视图，而不是不断增长的回滚。

每个框架都有两个部分，按以下顺序：

1. **当前状态**，打印精美且差异友好（每行一个字段，或格式化 JSON）。对字段名称或节标题使用 **粗体**，对不太重要的上下文（时间戳、ID、派生值）使用 **暗色**。原生 ANSI 转义码很好 — `\x1b[1m` 粗体，`\x1b[2m` 暗淡，`\x1b[0m` 重置。除非项目中已有样式库，否则无需引入样式库。
2. **键盘快捷键**，列于底部：`[a] 添加用户 [d] 删除用户 [t] 滴答时钟 [q] 退出`。将关键部分加粗，将描述变暗，反之亦然——只要读起来清晰即可。

行为：

1. **初始化状态** — 单个内存中对象/结构。开始时渲染第一帧。
2. **一次读取一个击键（或一行）**，分派到改变状态的处理程序。
3. **在每个动作之后重新渲染**全帧 - 不要追加、替换。
4. **循环直到退出。**整个框架应该适合一个屏幕。

### 5. 使其可以通过一个命令运行

将脚本添加到项目的现有任务运行程序（“package.json”脚本、“Makefile”、“justfile”、“pyproject.toml”）。用户应该运行“pnpm run <prototype-name>”或等效命令——永远不需要记住路径。

如果宿主项目没有任务运行程序，只需将命令放在原型自述文件的顶部即可。

### 6. 交接

向用户提供运行命令。他们会自己驾驶；有趣的时刻是当他们说“等等，这不应该是可能的”或“嗯，我以为 X 会不同”时——这些都是_idea_中的错误，这就是重点。如果他们想要添加新操作，请添加它们。原型不断发展。

### 7. 捕捉答案

当原型完成其工作后，问题的答案就是唯一值得保留的东西。如果用户在附近，询问它教会了他们什么。如果没有，请在原型旁边留下“NOTES.md”，以便在原型被删除之前填写答案（或者由您填写，如果您观看了会议）。

## 反模式

- **不要添加测试。** 需要测试的原型不再是原型。
- **不要将其连接到真实数据库。** 使用内存存储，除非问题专门与持久性有关。
- **不要一概而论。** 没有“如果我们以后想支持 X 该怎么办。”原型回答了一个问题。
- **不要将逻辑和 TUI 混淆在一起。** 如果减速器/状态机引用 `console.log`、提示或终端转义码，则它不再是可移植的。将 TUI 作为纯模块上的薄壳。
- **不要将 TUI shell 投入生产。** 该 shell 针对从终端手动驱动进行了优化。它背后的逻辑模块是值得保留的。
