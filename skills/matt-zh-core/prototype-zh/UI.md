# 用户界面原型

在单个路线上生成 **几种完全不同的 UI 变体**，可从浮动底部栏进行切换。用户在浏览器中切换变体，选择一个（或从每个变体中窃取一些内容），然后扔掉其余的。

如果问题是关于逻辑/状态而不是某个东西看起来像什么——错误的分支。使用 [LOGIC.md](LOGIC.md)。

## 当这是正确的形状时

- “这个页面应该是什么样子？”
- “我想在提交之前查看此仪表板的一些选项。”
- “尝试设置屏幕的不同布局。”
- 任何时候用户都会花一天时间在脑海中的三个模糊模型之间进行选择。

## 两个子形状——强烈推荐子形状 A

当 UI 原型与应用程序的其余部分相对应时，它更容易判断——真实的标题、真实的侧边栏、真实的数据、真实的密度。一次性路线本身就是一个真空：每个变体单独看起来都很好。只要存在合理的现有页面来托管变体，就默认为子形状 A。仅当原型附近确实没有家时才使用子形状 B。

### 子形状 A — 对现有页面的调整（首选）

该路线已经存在。变体在**相同的路线**上呈现，由“?variant=” URL 搜索参数控制。现有的数据获取、参数和身份验证都保留——只有渲染交换。这是默认值；除非有特定原因不选择，否则请选择它。

如果原型用于尚无页面但“自然会存在于页面内”的内容（仪表板的新部分、设置屏幕上的新卡、现有流程中的新步骤）——这仍然是子形状 A。将变体安装在主页内。

### 子形状 B — 新页面（最后的手段）

仅当正在原型化的东西确实没有现有页面可容纳时才使用此选项 - 例如一个全新的顶层表面，或者一个无法嵌入到任何合理位置的流。

按照项目已使用的任何路由约定创建**一次性路由** - 不要发明新的顶级结构。命名它，使其显然是一个原型（例如，在路径或文件名中包含“prototype”一词）。相同的“?variant=”模式。

在提交子形状 B 之前，请进行健全性检查：是否真的没有可以嵌入的现有页面？一条空的路线隐藏了一些设计问题，而有人居住的路线则会暴露这些问题。

在两个子形状中，浮动底栏是相同的。

## 流程

### 1. 陈述问题并选择 N

默认为 **3 个变体**。超过 5 个就不再完全不同，并开始成为噪音——仅限于此。

在原型位置或文件顶部注释中用一行写下该计划：

> “设置页面的三种变体，可在现有的‘/settings’路径上通过‘?variant=’进行切换。”

无论用户是否在这里进行反击，这都有效。

### 2. 生成完全不同的变体

起草每个变体。握住每一个以：

- 页面的用途及其有权访问的数据。
- 项目的组件库/样式系统（TailwindCSS、shadcn、MUI、纯 CSS 等）。
- 清晰的导出组件名称，例如“变体A”、“变体B”、“变体C”。

变体必须**结构上不同**——不同的布局、不同的信息层次、不同的主要可供性，而不仅仅是不同的颜色。三个稍微调整过的卡片网格不是 UI 原型，而是壁纸。如果两份草稿太相似，请重做一份并明确“不要使用卡片网格”指导。

### 3. 将它们连接在一起

在路由上创建单个切换器组件：
```tsx
// pseudo-code — adapt to the project's framework
const variant = searchParams.get('variant') ?? 'A';
return (
  <>
    {variant === 'A' && <VariantA {...data} />}
    {variant === 'B' && <VariantB {...data} />}
    {variant === 'C' && <VariantC {...data} />}
    <PrototypeSwitcher variants={['A','B','C']} current={variant} />
  </>
);
```
对于子形状A（现有页面）：将所有现有数据的获取保留在切换器上方；每个变体仅渲染的子树发生变化。

对于子形状 B（新页面）：`/prototype-zh/<name>` 下的一次性路由安装相同的切换器。

### 4. 构建浮动切换器

屏幕底部中央有一个固定位置的小栏，由三部分组成：

- **左箭头** — 循环到上一个变体（环绕）。
- **变体标签** — 显示当前变体键，如果变体导出名称，则也显示该名称。例如`B — 侧边栏布局`。
- **右箭头** — 向前循环（环绕）。

行为：

- 单击箭头更新 URL 搜索参数（使用框架的路由器 - Next 上的“router.replace”，React Router 上的“navigate”等），因此该变体是可共享的并且重新加载稳定。
- 键盘：`←`和`→`箭头键也循环。当“<input>”、“<textarea>”或“[contenteditable]”获得焦点时，不要拦截箭头键。
- 视觉上与页面不同（例如高对比度药丸、微妙的阴影），因此它显然不属于正在评估的设计的一部分。
- 隐藏在生产版本中 - `process.env.NODE_ENV !== '生产'` 上的门或等效检查，因此杂散原型合并无法将栏发送给用户。

将切换器放在单个共享组件中，以便两个子形状都可以重用它。将其定位到项目中共享 UI 所在的位置。

### 5. 移交

显示 URL（和“?variant=”键）。用户只要到达它就会翻阅它。有趣的反馈通常是**“我想要 B 的标题和 C 的侧边栏”** - 这就是他们想要的实际设计。

### 6. 捕获答案并清理

一旦某个变体获胜，记下哪个变体以及原因（提交消息、ADR、问题，或者如果运行 AFK 并且用户尚未响应，则在原型旁边添加“NOTES.md”）。然后：

- **子形状 A** — 删除丢失的变体和切换器；将获胜者折叠到现有页面中。
- **子形状 B** — 将获胜变体提升为真实路线，删除一次性路线和切换器。

不要将不同的组件或切换器随意放置。它们很快就会腐烂，让下一个读者感到困惑。

## 反模式

- **仅在颜色或副本上有所不同的变体。**这是一个调整，而不是原型。真正的变体在结构上不一致。
- **在变体之间共享太多代码。** 共享“<Header>”就可以了；共享的“<Layout>”违背了这一点。每个变体都应该可以自由地抛弃布局。
- **将变体连接到真正的突变。** 只读原型就可以了。如果一个变体需要变异，请将其指向一个存根 - 问题是“这应该是什么样子”，而不是“后端是否工作”。
- **将原型直接推广到生产。** 变体代码是在原型约束下编写的（无测试，最少的错误处理）。当你折叠它时，正确地重写它。
