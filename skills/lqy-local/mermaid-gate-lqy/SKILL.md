# Mermaid 设计图 Gate

`$to-spec-lqy` 写 spec 时引用它记录上级判断；`$to-tickets-lqy` 规划 Ticket 时引用它记录每个 Ticket 的局部判断；`$implement-lqy` 开始实现前用它检查当前 Ticket 是否已经满足。

## Gate

先判断当前 Ticket 是否需要 Mermaid 图。

- **需要图**：会改变架构边界、公共接口、adapter contract、状态机、主要调用流程或跨模块依赖。
- **不需要图**：纯文档、调研记录、文案或无接口变化的小修。

这是一个 gate，不是每个任务都强制画图。需要图时，必须先产出 8 张图；不需要图时，必须在 Ticket、issue comment 或 PR 描述中说明原因。

## 必须的 8 张图

需要图时，先使用 `/mermaid-visualizer`，并把图放到 issue 正文、issue comment 或 PR 描述中，再开始实现。

必须画 4 类图，每类都包含 **当前** 和 **目标** 两版，共 8 张：

- **架构/调用关系图**：模块边界、接口、adapter、跨模块依赖和调用方向。
- **时序图**：一次请求、一次工具调用或一次业务流程的步骤顺序。
- **状态图**：状态机、生命周期、任务状态或可回退流程。
- **类图**：当前 Ticket 涉及的局部类/接口结构、职责和依赖；避免画全项目类图。

如果当前没有对应结构，也要画“当前”图，并明确标注当前不存在该结构。

## Ticket 验收标准写法

需要图时，Ticket 的 acceptance criteria 只写交付物，不要重复 Gate 判断条件。推荐写法：

- [ ] 需要补齐 8 张 Mermaid 图：架构/调用关系、时序、状态、类图各包含当前和目标两版，并说明变化影响。

不要把“如涉及架构边界、公共接口、adapter contract、状态机、主要调用流程或跨模块依赖”这类判断条件写进 TODO；这些判断只属于本 Gate。

## 更新规则

实现中如果接口、状态机、依赖方向或调用流程与图不一致，先更新图，再继续实现。完成前确认保留的图反映最终实现。

## 调用方式

调用方只需要引用 `$mermaid-gate-lqy`，不要在通用 Ticket 模板里重复本 skill 的细则。AI 应按本 skill 判断当前 Ticket 是否需要图，并在需要时补齐 8 张图；不需要图时说明原因。
