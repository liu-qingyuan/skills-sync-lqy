# 代码库设计

设计**深模块**：用小接口隐藏大量行为，把接口放在清晰的接缝上，并通过这个接口测试模块。只要在设计或重构代码，就使用这套语言和原则。目标是让调用方获得杠杆，让维护者获得局部性，并让代码易于测试。

## 术语表

严格使用这些术语——不要替换成“组件”、“服务”、“API”或“边界”。保持语言一致正是这套词汇的意义所在。

**Module（模块）** — 任何拥有接口和实现的东西。这个词刻意不绑定规模：它可以是函数、类、包，也可以是跨越多层的一条切片。_避免_：单元、组件、服务。

**Interface（接口）** — 调用方为了正确使用模块而必须知道的一切：不只是类型签名，还包括不变量、调用顺序约束、错误模式、必要配置和性能特征。_避免_：API、签名（太窄——它们只描述类型层面的表面）。

**Implementation（实现）** — 模块内部的东西，也就是它的代码主体。它不同于 **Adapter**：一个东西可以是小适配器但有大量实现（比如 Postgres repository），也可以是大适配器但实现很少（比如内存 fake）。讨论接缝时用“适配器”；其他时候用“实现”。

**Depth（深度）** — 接口上的杠杆：调用方（或测试）每学习一单位接口，能驱动多少行为。大量行为藏在小接口后面时，模块就是**深**的；接口几乎和实现一样复杂时，模块就是**浅**的。

**Seam（接缝）** _(Michael Feathers)_ — 一个可以改变行为、但不必在该处编辑代码的位置；也就是模块接口所在的*位置*。接缝放在哪里本身就是一个设计决策，和接缝背后放什么是两件事。_避免_：边界（容易和 DDD 的限界上下文混在一起）。

**Adapter（适配器）** — 在某个接缝处满足接口的具体事物。它描述的是*角色*（填哪个槽位），不是材质（里面是什么）。

**Leverage（杠杆）** — 调用方从深度中得到的收益：每学习一单位接口，就获得更多能力。一个实现可以在 N 个调用点和 M 个测试中持续回本。

**Locality（局部性）** — 维护者从深度中得到的收益：变化、缺陷、知识和验证集中在一个地方，而不是散落到各个调用方。修一次，到处都修好。

## 深与浅

**深模块** = 小接口 + 大量实现：

```
┌─────────────────────┐
│   Small Interface   │  ← Few methods, simple params
├─────────────────────┤
│                     │
│  Deep Implementation│  ← Complex logic hidden
│                     │
└─────────────────────┘
```

**浅模块** = 大接口 + 少量实现（应避免）：

```
┌─────────────────────────────────┐
│       Large Interface           │  ← Many methods, complex params
├─────────────────────────────────┤
│  Thin Implementation            │  ← Just passes through
└─────────────────────────────────┘
```

设计接口时，问自己：

- 能不能减少方法数量？
- 能不能简化参数？
- 能不能把更多复杂性藏到里面？

## 原则

- **深度是接口的属性，不是实现的属性。** 深模块内部可以由小的、可 mock、可替换的部分组成——只是这些部分不属于接口。模块既可以有**内部接缝**（实现私有、供自身测试使用），也可以在接口处有**外部接缝**。
- **删除测试。** 想象把这个模块删掉。如果复杂性也消失了，它原本只是透传层。如果复杂性会重新出现在 N 个调用方里，它就在创造价值。
- **接口就是测试表面。** 调用方和测试跨过的是同一条接缝。如果你想越过接口去测试，模块形状很可能不对。
- **一个适配器意味着接缝还只是设想；两个适配器才说明接缝真实存在。** 除非接缝两侧确实有会变化的东西，否则不要引入接缝。

## 为可测试性设计

好的接口会让测试变得自然：

1. **接收依赖，不要在内部创建依赖。**

   ```typescript
   // Testable
   function processOrder(order, paymentGateway) {}

   // Hard to test
   function processOrder(order) {
     const gateway = new StripeGateway();
   }
   ```

2. **返回结果，不要直接制造副作用。**

   ```typescript
   // Testable
   function calculateDiscount(cart): Discount {}

   // Hard to test
   function applyDiscount(cart): void {
     cart.total -= discount;
   }
   ```

3. **缩小表面积。** 方法更少 = 需要的测试更少。参数更少 = 测试准备更简单。

## 关系

- 一个 **Module（模块）** 正好有一个 **Interface（接口）**（它呈现给调用方和测试的表面）。
- **Depth（深度）** 是 **Module（模块）** 的属性，要相对于它的 **Interface（接口）** 来衡量。
- **Seam（接缝）** 是 **Module（模块）** 的 **Interface（接口）** 所在的位置。
- **Adapter（适配器）** 位于 **Seam（接缝）** 上，并满足 **Interface（接口）**。
- **Depth（深度）** 为调用方产生 **Leverage（杠杆）**，为维护者产生 **Locality（局部性）**。

## 拒绝的框架

- **把深度看作实现行数与接口行数的比例**（Ousterhout）：这会奖励给实现灌水。这里采用的是“深度即杠杆”。
- **把“Interface”理解成 TypeScript 的 `interface` 关键字或类的 public 方法**：太窄——这里的接口包含调用方必须知道的所有事实。
- **“边界”**：容易和 DDD 的限界上下文混在一起。请说**接缝**或**接口**。
