# 诊断错误

针对硬错误的纪律。仅当明确合理时才跳过阶段。

在探索代码库时，请阅读“CONTEXT.md”（如果存在）以获得相关模块的清晰思维模型，并检查您正在接触的区域中的 ADR。

## 第 1 阶段 — 建立反馈循环

**这就是技巧。** 其他一切都是机械的。如果您对该错误有一个**严格的**通过/失败信号（该信号在_this_错误上变红），您就会找到原因；二分法、假设检验和仪器仪表都只会消耗它。如果你没有，再多的盯着代码也救不了你。

在这里花费不成比例的努力。 **要有侵略性。要有创意。拒绝放弃。**

### 构建方法 - 大致按以下顺序尝试

1. **测试失败**在任何接缝处都会出现错误 - 单元、集成、e2e。
2. **Curl / HTTP 脚本** 针对正在运行的开发服务器。
3. **CLI 调用** 使用固定输入，将标准输出与已知良好的快照进行比较。
4. **无头浏览器脚本** (Playwright / Puppeteer) — 驱动 UI，在 DOM/控制台/网络上进行断言。
5. **重放捕获的跟踪。** 将真实的网络请求/负载/事件日志保存到磁盘；通过隔离的代码路径重播它。
6. **一次性工具。** 启动系统的最小子集（一个服务，模拟的 deps），通过单个函数调用来执行错误代码路径。
7. **属性/模糊循环。** 如果 bug 是“有时输出错误”，则运行 1000 个随机输入并查找故障模式。
8. **二分法。** 如果错误出现在两个已知状态（提交、数据集、版本）之间，请自动“在状态 X 启动，检查，重复”，以便您可以“git bisect run”它。
9. **差异循环。** 通过旧版本与新版本（或两个配置）运行相同的输入并比较输出。
10. **HITL bash 脚本。** 最后的手段。如果人类必须点击，请使用“scripts/hitl-loop.template.sh”驱动它们，以便循环仍然是结构化的。捕获的输出反馈给您。

建立正确的反馈循环，bug 就被修复了 90%。

### 拉紧循环

将循环视为产品。一旦你有了_a_循环，**收紧**它：

- 我可以让它更快吗？ （缓存设置，跳过不相关的init，缩小测试范围。）
- 我可以让信号更清晰吗？ （断言特定症状，而不是“没有崩溃”。）
- 我可以让它更加确定吗？ （固定时间、种子 RNG、隔离文件系统、冻结网络。）

30 秒的片状循环仅仅比没有循环好一点； 2 秒的确定性是严格的——一种调试超能力。

### 非确定性错误

我们的目标不是干净的复制，而是**更高的复制率**。将触发器循环 100 倍、并行化、增加压力、缩小计时窗口、注入睡眠。 50% 的错误是可调试的； 1% 不是——继续提高速率，直到可以调试为止。

### 当你确实无法构建循环时

停下来并明确地说出来。列出您尝试过的内容。要求用户：(a) 访问重现它的任何环境，(b) 捕获的工件（HAR 文件、日志转储、核心转储、带有时间戳的屏幕记录），或 (c) 添加临时生产工具的权限。 **不要**在没有循环的情况下进行假设。

### 完成标准 — 紧密循环变红

当循环**紧密**并且**具有红色能力**时，阶段 1 就完成了：您可以命名**一个命令** - 脚本路径、测试调用、curl - 您已经**至少运行过一次**（粘贴调用及其输出），即：- [ ] **红色功能** - 它驱动实际的错误代码路径并断言**用户的确切症状**，因此它可以在此错误上变为红色，并在修复后变为绿色。不是“运行无错误”——它必须能够_捕获这个特定的错误_。
- [ ] **确定性** — 每次运行都会得出相同的结论（片状错误：固定的、高再现率，如上所述）。
- [ ] **快** — 秒，而不是分钟。
- [ ] **agent 可运行** — 您可以在无人值守的情况下运行它；人类只能通过“scripts/hitl-loop.template.sh”参与循环。

如果您发现自己在该命令存在之前正在阅读代码来构建理论，**停止 - 直接跳到假设正是该技能可以防止的失败。**没有红色功能的命令，没有阶段 2。

## 第 2 阶段 — 再现 + 最小化

运行循环。看着它变红——错误出现了。

确认：

- [ ] 循环产生 **用户** 描述的故障模式 - 不是碰巧在附近的不同故障。错误的错误=错误的修复。
- [ ] 故障可以在多次运行中重现（或者，对于非确定性错误，可以以足够高的速率重现以进行调试）。
- [ ] 您已经捕获了确切的症状（错误消息、错误输出、缓慢的计时），因此后续阶段可以验证修复是否确实解决了该问题。

### 最小化

一旦它变红，将重现缩小到**仍然变红的最小场景**。削减输入、调用者、配置、数据和步骤**一次一个**，每次削减后重新运行循环 - 仅保留故障的承载部分。

为什么要费心：最小的重现缩小了第 3 阶段的假设空间（留下可疑的移动部件更少），并成为第 5 阶段的干净回归测试。

当**每个剩余元素都是承重**时完成 - 删除其中任何一个都会使循环变绿。

在复制**和**最小化之前不要继续。

## 第三阶段——假设

在测试其中任何一个之前，生成**3-5 个排名假设**。单一假设的生成基于第一个看似合理的想法。

每个假设都必须是**可证伪的**：说明它所做的预测。

> 格式：“如果<X>是原因，那么<更改Y>将使错误消失/<更改Z>会使情况变得更糟。”

如果你无法说出预测，那么假设只是一种氛围——丢弃或强化它。

**在测试之前向用户显示排名列表。** 他们通常拥有立即重新排名的领域知识（“我们刚刚部署了对 #3 的更改”），或者知道他们已经排除的假设。便宜的检查站，节省大量时间。不要阻止它——如果用户 AFK，则继续进行排名。

## 第 4 阶段 — 仪器

每个探针必须映射到第 3 阶段的特定预测。 **一次更改一个变量。**

工具偏好：

1. **调试器/REPL 检查**（如果环境支持）。一个断点胜过十个日志。
2.**有针对性的日志**在区分假设的边界处。
3. 永远不要“记录所有内容并 grep”。

**使用唯一的前缀标记每个调试日志**，例如`[调试-a4f2]`。最后的清理工作变成了一个 grep。未标记的日志得以保存；标记的日志死亡。

**性能分支。** 对于性能回归，日志通常是错误的。相反：建立基线测量（时序工具、“performance.now()”、分析器、查询计划），然后一分为二。首先测量，其次修复。

## 阶段 5 — 修复 + 回归测试

**在修复之前**编写回归测试——但前提是存在**正确的接缝**。

正确的接缝是测试在调用站点发生时执行**真实错误模式**的接缝。如果唯一可用的接缝太浅（当错误需要多个调用者时进行单调用者测试，无法复制触发错误的链的单元测试），则那里的回归测试会给出错误的置信度。**如果不存在正确的接缝，这本身就是发现的结果。**注意它。代码库架构正在防止错误被锁定。将其标记为下一阶段。

如果存在正确的接缝：

1. 将最小化的重现转化为该接缝处的失败测试。
2. 看着它失败。
3. 应用修复。
4. 看着它过去。
5. 针对原始（未最小化）场景重新运行第一阶段反馈循环。

## 第 6 阶段 — 清理 + 事后分析

声明完成之前需要：

- [ ] 原始重现不再重现（重新运行第一阶段循环）
- [ ] 回归测试通过（或记录了不存在接缝）
- [ ] 所有 `[DEBUG-...]` 检测已删除（`grep` 前缀）
- [ ] 删除了一次性原型（或移动到明确标记的调试位置）
- [ ] 事实证明正确的假设在提交/PR 消息中进行了说明——以便下一个调试器学习

**然后问：什么可以防止这个错误？** 如果答案涉及架构更改（没有好的测试缝、混乱的调用者、隐藏的耦合），则将具体细节交给“/improve-codebase-architecture-lqy”技能。在修复完成之后（而不是之前）提出建议 — 您现在比开始时拥有更多信息。
