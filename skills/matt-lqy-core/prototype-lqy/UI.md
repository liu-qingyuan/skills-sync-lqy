# UI 原型

在单一路由上生成**几个完全不同的 UI 变体**，并通过浮动底栏切换。用户在浏览器里来回切换，选中一个方案（或从多个方案里各取一部分），然后丢弃其余部分。

如果问题关注的是逻辑/状态，而不是界面长什么样——那就是选错分支。使用 [LOGIC.md](LOGIC.md)。

## 什么时候适合这种形态

- “这个页面应该长什么样？”
- “提交前我想看几个仪表盘方案。”
- “试试设置页的不同布局。”
- 用户可能会花一整天在脑中三个模糊模型之间摇摆时。

## 两种子形态——强烈优先选择子形态 A

当 UI 原型能放进应用其他部分时，更容易判断好坏——真实的标题、真实的侧边栏、真实的数据、真实的密度。孤立的一次性路由像真空环境：每个变体单看都可能不错。只要有合理的现有页面能承载这些变体，就默认选择子形态 A。只有原型确实没有合适归属时，才使用子形态 B。

### 子形态 A — 调整现有页面（首选）

路由已经存在。变体在**同一个路由**上渲染，由 `?variant=` URL 搜索参数控制。保留现有的数据获取、参数和认证——只替换渲染部分。这是默认选择；除非有明确理由，否则就用它。

如果原型针对的是尚无独立页面、但“自然会存在于某个页面内”的内容（仪表盘的新区域、设置页的新卡片、现有流程里的新步骤）——这仍然是子形态 A。把变体安装到宿主页内。

### 子形态 B — 新页面（最后手段）

只有当正在原型化的界面确实没有现有页面可容纳时，才使用这个选项——例如全新的顶层界面，或无法嵌入任何合理位置的流程。

按项目已有路由约定创建**一次性路由**——不要发明新的顶层结构。命名要明显表明它是原型（例如在路径或文件名中包含 `prototype`）。同样使用 `?variant=` 模式。

在决定采用子形态 B 前，先做一次合理性检查：真的没有任何现有页面可以嵌入它吗？空白路由会掩盖设计问题，而带真实内容的路由会暴露这些问题。

两种子形态都使用同一个浮动底栏。

## 流程

### 1. 写明问题并选择 N

默认生成 **3 个变体**。超过 5 个后就很难保持真正不同，反而会变成噪音——最多 5 个。

在原型位置或文件顶部注释里，用一行写下计划：

> “设置页的三个变体，在现有 `/settings` 路由上通过 `?variant=` 切换。”

无论用户是否在线并会提出异议，这一步都有效。

### 2. 生成完全不同的变体

起草每个变体。每个变体都要受以下约束：

- 页面的用途，以及它能访问的数据。
- 项目的组件库 / 样式系统（TailwindCSS、shadcn、MUI、纯 CSS 等）。
- 清晰的导出组件名，例如 `VariantA`、`VariantB`、`VariantC`。

变体必须**结构上不同**——布局不同、信息层级不同、主要操作入口不同，而不只是颜色不同。三个略微调整过的卡片网格不是 UI 原型，只是壁纸。如果两个草稿太像，重做其中一个，并明确给出“不要使用卡片网格”的指导。

### 3. 把它们连接起来

在路由上创建一个切换组件：

```tsx
// pseudo-code — adapt to the project's framework
const variant = searchParams.get('variant') ?? 'A';
return (
  <>
    {variant === 'A' && <VariantA {...data} />}
    {variant === 'B' && <VariantB {...data} />}
    {variant === 'C' && <VariantC {...data} />}
    <PrototypeSwitcher variants={['A','B','C']} current={variant} />
  </>
);
```

对于子形态 A（现有页面）：把所有现有数据获取保留在切换器上方；每个变体只改变被渲染的子树。

对于子形态 B（新页面）：`/prototype/<name>` 下的一次性路由挂载同一个切换器。

### 4. 构建浮动切换器

在屏幕底部中央放一个小型 fixed 定位栏，包含三部分：

- **左箭头** — 循环到上一个变体（到头后回绕）。
- **变体标签** — 显示当前变体键；如果变体导出了名称，也一起显示。例如 `B — Sidebar layout`。
- **右箭头** — 循环到下一个变体（到头后回绕）。

行为：

- 点击箭头会更新 URL 搜索参数（使用框架路由器——Next 用 `router.replace`，React Router 用 `navigate` 等），这样变体链接可分享，刷新后也稳定。
- 键盘：`←` 和 `→` 方向键也能循环切换。当焦点在 `<input>`、`<textarea>` 或 `[contenteditable]` 上时，不要拦截方向键。
- 视觉上要和页面区分开（例如高对比度胶囊形、轻微阴影），让人明显知道它不是正在评估的设计本身。
- 生产构建中隐藏——用 `process.env.NODE_ENV !== 'production'` 或等价检查做门禁，避免误合并的原型把切换栏发给用户。

把切换器放在单个共享组件中，让两种子形态都能复用。位置放在项目共享 UI 通常所在的地方。

### 5. 交接

给出 URL（以及 `?variant=` 的键）。用户有空时会自己翻看。最有价值的反馈通常是**“我想要 B 的标题，再加上 C 的侧边栏”**——那才是他们真正想要的设计。

### 6. 捕获答案并清理

一旦某个变体胜出，记录胜出的是哪个以及原因（提交信息、ADR、issue；如果用户不在线，则在原型旁边放一个 `NOTES.md`）。然后：

- **子形态 A** — 删除落选变体和切换器；把胜出方案折叠进现有页面。
- **子形态 B** — 把胜出变体提升为真实路由，删除一次性路由和切换器。

不要把变体组件或切换器随手留在仓库里。它们很快会腐烂，并让下一个读者困惑。

## 反模式

- **只在颜色或文案上不同的变体。** 那只是微调，不是原型。真正的变体在结构上有分歧。
- **变体之间共享太多代码。** 共享 `<Header>` 可以；共享 `<Layout>` 会破坏原型目的。每个变体都应该可以自由抛弃布局。
- **把变体接到真实写操作上。** 只读原型可以。如果某个变体需要写入，就指向 stub——问题是“它应该长什么样”，不是“后端能不能工作”。
- **把原型直接推上生产。** 变体代码是在原型约束下写的（无测试、最少错误处理）。折叠进真实代码时，要认真重写。
