# 原型

原型是**用一次性代码回答一个问题**。问题决定原型的形态。

## 选择一个分支

先判断你要回答哪个问题——根据用户提示、周围代码判断；如果用户在线，也可以直接询问：

- **“这个逻辑/状态模型对吗？”** → [LOGIC.md](LOGIC.md)。构建一个小型交互式终端 app，手动推动状态机，覆盖纸面上难以推演的情况。
- **“这应该长什么样？”** → [UI.md](UI.md)。在单一路由上生成几个完全不同的 UI 变体，并通过 URL 搜索参数和浮动底栏切换。

这两个分支会产出完全不同的工件——选错分支会让整个原型白做。如果问题确实不清楚，且联系不上用户，就默认选择更贴近周围代码的分支（后端模块 → 逻辑；页面或组件 → UI），并在原型顶部写明这个假设。

## 两个分支都适用的规则

1. **从一开始就按可丢弃代码处理，并清楚标记。** 把原型代码放在实际使用位置附近（靠近它正在原型化的模块或页面），这样上下文一眼可见；但命名要让普通读者看得出这是原型，不是生产代码。一次性 UI 路由要遵守项目已有路由约定；不要发明新的顶层结构。
2. **一个命令即可运行。** 使用项目现有任务运行方式支持的命令——`pnpm <name>`、`python <path>`、`bun <path>` 等。用户应该不用思考就能启动它。
3. **默认不要持久化。** 状态保存在内存中。持久化是原型要_检验_的内容，不是它应该依赖的内容。如果问题明确涉及数据库，就使用临时数据库，或使用带有明确 `PROTOTYPE — erase me` 名称的本地文件。
4. **跳过打磨。** 除了让原型_能运行_，不要写测试、错误处理或抽象。重点是快速学到结论，然后删除它。
5. **展示状态。** 每次操作（逻辑）或每次切换变体（UI）后，打印或渲染完整的相关状态，让用户看清发生了什么变化。
6. **结束后删除或吸收。** 原型回答完问题后，要么删除它，要么把验证过的决策折叠进真实代码——不要让它留在仓库里慢慢腐烂。

## 完成后

原型里唯一值得保留的是_答案_。把答案和它回答的问题一起记录到持久位置：提交信息、ADR、issue，或原型旁边的 `NOTES.md`。如果用户在线，快速对话即可完成记录；如果不在线，就留下占位，等他们（或下一轮的你）在删除原型前补上结论。
