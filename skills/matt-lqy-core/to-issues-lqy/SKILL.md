# 问题

使用垂直切片（示踪子弹）将计划分解为可独立抓取的问题。

issue tracker 和triage 标签词汇表应该已提供给您 - 如果没有，请运行`$setup-matt-pocock-skills-lqy`。

## 流程

### 1. 收集背景信息

从对话上下文中已有的内容开始工作。如果用户将问题引用（问题编号、URL 或路径）作为参数传递，请从 issue tracker 中获取它并阅读其完整正文和评论。

### 2. 探索代码库（可选）

如果您尚未探索代码库，请这样做以了解代码的当前状态。问题标题和描述应使用项目的领域词汇表，并尊重您所涉及领域的 ADR。

寻找机会预先分解代码以使实施变得更容易。 “让改变变得容易，然后让改变变得容易。”

### 3. 绘制垂直切片

将计划分解为**示踪子弹**问题。每个问题都是一个薄的垂直切片，端到端地贯穿所有集成层，而不是一层的水平切片。

<垂直切片规则>

- 每个切片都提供一条狭窄但完整的路径穿过每一层（架构、API、UI、测试）
- 完整的切片可以自行拆卸或验证
- 任何预分解都应该首先完成

</垂直切片规则>

### 4. 对用户进行测验

将提议的细目分类呈现为编号列表。对于每个切片，显示：

- **标题**：简短的描述性名称
- **被**阻止：其他切片（如果有）必须首先完成
- **涵盖的用户故事**：此内容涉及哪些用户故事（如果源材料有）

询问用户：

- 粒度感觉合适吗？ （太粗/太细）
- 依赖关系是否正确？
- 是否应该合并或进一步分割任何切片？

迭代直到用户批准细分。

### 5. 将问题发布到 issue tracker

对于每个批准的切片，将新 issue 发布到 issue tracker。使用下面的 issue 正文模板。这些问题被认为已为 AFK agent 做好准备，因此除非另有说明，否则请使用正确的triage 标签发布它们。

按依赖顺序发布问题（首先是阻止者），以便您可以在“阻止者”字段中引用实际问题标识符。

<问题模板>
## Parent issue

对 issue tracker 上父级 issue 的引用（如果来源是现有 issue，否则忽略此部分）。

## 构建什么

这个垂直切片的简洁描述。描述端到端的行为，而不是逐层的实现。

避免使用特定的文件路径或代码片段——它们很快就会过时。例外：如果原型生成的代码片段比散文更精确地编码决策（状态机、reducer、模式、类型形状），请将其内联到此处并简要说明它来自原型。修剪决策丰富的部分——不是工作演示，只是重要的部分。

## 验收标准

- [ ] 标准 1
- [ ] 标准 2
- [ ] 标准 3

## 被阻止

- 阻止票的参考（如果有）

或者如果没有阻止程序，则“无 - 可以立即开始”。

</问题模板>

不要关闭或修改任何 parent issue。
