# 领域建模

在设计过程中主动构建并打磨项目的领域模型。这是一项主动工作：质疑术语、设计边界场景，并在术语和决策刚刚明确时立刻写入术语表和决策记录。（只是读取 `CONTEXT.md` 来获取词汇，不算使用这个 skill；那只是任何 skill 都可以遵守的一条习惯。本 skill 用于改变领域模型，而不是单纯消费领域模型。）

## 文件结构

大多数仓库只有一个领域上下文：

```text
/
├── CONTEXT.md
├── docs/
│   └── adr/
│       ├── 0001-event-sourced-orders.md
│       └── 0002-postgres-for-write-model.md
└── src/
```

如果仓库根目录存在 `CONTEXT-MAP.md`，说明该仓库有多个领域上下文。这个 map 会列出每个上下文所在的位置：

```text
/
├── CONTEXT-MAP.md
├── docs/
│   └── adr/                          ← 系统级决策
├── src/
│   ├── ordering/
│   │   ├── CONTEXT.md
│   │   └── docs/adr/                 ← 特定上下文的决策
│   └── billing/
│       ├── CONTEXT.md
│       └── docs/adr/
```

按需创建文件：只有真正有内容要写时才创建。若还没有 `CONTEXT.md`，就在第一个术语被明确时创建；若还没有 `docs/adr/`，就在第一个 ADR 被需要时创建。

## 会话中怎么做

### 对照术语表提出质疑

当用户使用的术语与 `CONTEXT.md` 中已有语言冲突时，立即指出：例如“术语表里把 cancellation 定义为 X，但你现在似乎在说 Y——你想表达的是哪一个？”

### 收紧模糊语言

当用户使用模糊或承载过多含义的术语时，提出一个更精确的规范术语。例如：“你说的 account 是指 Customer 还是 User？这两个概念不同。”

### 讨论具体场景

讨论领域关系时，用具体场景做压力测试。主动构造边界场景，逼出概念之间的精确边界。

### 提问前先查项目

每次准备向用户提出领域问题前，先判断答案是否能从当前项目中查明。若阅读本地代码、测试、配置、`CONTEXT.md`、ADR，或使用可用且最新的 `$gitnexus` 就能确定当前行为、术语或边界，先自行调查；GitNexus 用于定位关系和候选路径，关键结论仍用源码和测试确认。不要把可以查证的项目现状反问给用户。

当用户的描述与现有实现不一致时，先区分描述的是当前系统还是未来目标：

- **当前系统**：直接说明代码实际如何工作，并指出描述与实现的差异；不要让用户裁定可由代码确认的事实。
- **未来目标**：明确指出计划中的规则与当前实现不同，再询问或确认是否要把现有行为改成该目标。只有这种功能、需求或领域规则变更需要用户做决策。

例如，代码当前只支持整单取消时，如果用户在说明现状，就直接纠正为“当前实现会取消整张订单”；如果用户希望支持部分取消，则确认“这是要把现有规则改为支持部分取消吗？”

### 及时更新 CONTEXT.md

术语一旦明确，就立刻更新 `CONTEXT.md`。不要攒到最后统一处理；结论出现时就捕获。格式见 [CONTEXT-FORMAT.md](./CONTEXT-FORMAT.md)。

`CONTEXT.md` 必须完全排除实现细节。不要把 `CONTEXT.md` 当规格说明、草稿本或实现决策记录；它只是术语表，仅此而已。

### 谨慎提出 ADR

只有同时满足以下三个条件时，才建议创建 ADR：

1. **难以逆转** — 以后改变决定的成本不低。
2. **缺少上下文会显得意外** — 未来读者会疑惑“为什么要这样做？”
3. **来自真实权衡** — 确实存在替代方案，而当前方案是出于具体理由被选中的。

只要缺少其中任何一个条件，就跳过 ADR。格式见 [ADR-FORMAT.md](./ADR-FORMAT.md)。
