# ADR 格式

ADR 放在 `docs/adr/` 下，并使用连续编号：`0001-slug.md`、`0002-slug.md` 等。

按需创建 `docs/adr/` 目录：只有第一个 ADR 真正需要写入时才创建。

## 模板

```md
# {决策短标题}

{1-3 句话：背景是什么、我们决定了什么、为什么这样决定。}
```

就这些。一个 ADR 可以只有一个段落。ADR 的价值在于记录“做过这个决定”以及“为什么做这个决定”，而不是填满一堆固定章节。

## 可选章节

只有在确实增加价值时才添加这些章节。多数 ADR 不需要它们。

- **Status** frontmatter（`proposed | accepted | deprecated | superseded by ADR-NNNN`）— 当决策可能被重新审视时有用。
- **Considered Options** — 只有被拒绝的替代方案值得未来记住时才写。
- **Consequences** — 只有存在不明显的下游影响时才写。

## 编号

扫描 `docs/adr/` 中已有的最大编号，并在此基础上加一。

## 什么时候建议写 ADR

必须同时满足以下三个条件：

1. **难以逆转** — 以后改变决定的成本不低。
2. **缺少上下文会显得意外** — 未来读者看到代码时会疑惑“为什么会这样做？”
3. **来自真实权衡** — 确实存在替代方案，而你出于具体理由选择了当前方案。

如果一个决定很容易回滚，就跳过；之后真要改也可以直接改。如果它并不意外，未来没人会追问原因。如果没有真实替代方案，也无需记录“我们做了显而易见的事”。

### 哪些内容适合写 ADR

- **架构形状。** 例如“我们使用 monorepo”，“写模型采用 event sourcing，读模型投影到 Postgres”。
- **上下文之间的集成模式。** 例如“Ordering 和 Billing 通过 domain event 通信，而不是同步 HTTP”。
- **带来锁定成本的技术选择。** 数据库、消息总线、认证提供商、部署目标等；不是每个库都要记，只记录那些替换成本很高的选择。
- **边界和范围决策。** 例如“Customer 数据归 Customer 上下文所有，其他上下文只能通过 ID 引用”。明确的“不做”与“要做”同样有价值。
- **有意偏离直觉方案。** 例如“我们不用 ORM，而使用手写 SQL，因为 X”。凡是合理读者可能默认选择相反方案的地方，都值得记录，避免下一位工程师把有意设计当成问题修掉。
- **代码里看不出来的约束。** 例如“合规要求禁止使用 AWS”，“合作伙伴 API 合同要求响应时间低于 200ms”。
- **拒绝理由不明显的替代方案。** 如果你认真考虑过 GraphQL，但因为细微原因选择 REST，就记录下来；否则六个月后还会有人再次提议 GraphQL。
