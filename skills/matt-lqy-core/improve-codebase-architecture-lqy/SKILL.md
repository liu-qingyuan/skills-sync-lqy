# 改进代码库架构

发现架构摩擦并提出**加深机会** - 将浅模块转变为深模块的重构。目标是可测试性和AI 可导航性。

该命令由项目的域模型_通知_并基于共享设计词汇构建：

- 针对架构词汇（**模块**、**接口**、**深度**、**接缝**、**适配器**、**杠杆**、**局部性**）及其原理（删除测试、“接口是测试面”、“一个适配器 = 假设接缝，两个 = 真实”）运行“/codebase-design-lqy”技能。在每个建议中准确使用这些术语——不要陷入“组件”、“服务”、“API”或“边界”。
- `CONTEXT.md` 中的领域语言为好接缝命名； `docs/adr/` 中的 ADR 记录了该命令不应重新提起诉讼的决定。

## 流程

### 1. 探索

阅读项目的领域术语表 (`CONTEXT.md`) 以及您首先接触的区域中的任何 ADR。

然后使用 Agent 工具和“subagent_type=Explore”来遍历代码库。不要遵循僵化的启发法——有机地探索并记下你在哪里遇到摩擦：

- 理解一个概念在哪里需要在许多小模块之间切换？
- 模块在哪里**浅**——接口几乎与实现一样复杂？
- 在哪里提取纯函数只是为了可测试性，但真正的错误隐藏在它们的调用方式中（没有**局部性**）？
- 紧密耦合的模块在哪里会从接缝处泄漏？
- 代码库的哪些部分未经测试，或者很难通过当前的界面进行测试？

将**删除测试**应用于您怀疑是浅层的任何内容：删除它会集中复杂性，还是只是移动它？ “是的，集中注意力”就是你想要的信号。

### 2. 以 HTML 报告形式呈现候选人

将一个独立的 HTML 文件写入操作系统临时目录，这样仓库中就不会出现任何内容。从“$TMPDIR”解析临时目录，回退到“/tmp”（或 Windows 上的“%TEMP%”），并写入“<tmpdir>/architecture-review-<timestamp>.html”，以便每次运行都会获得一个新文件。为用户打开它 - Linux 上的“xdg-open <path>”、macOS 上的“open <path>”、Windows 上的“start <path>” - 并告诉他们绝对路径。

该报告使用 **Tailwind via CDN** 进行布局和样式设置，并使用 **Mermaid via CDN** 进行图表，其中图形/流程/序列可靠地传达了结构。将 Mermaid 与手工制作的 CSS/SVG 视觉效果混合在一起 — 当关系是图形形状（调用图、依赖关系、序列）时使用 Mermaid，当您想要更多编辑效果（质量图、横截面、折叠动画）时使用手工构建的 div/SVG。每个候选人都会获得**之前/之后的可视化**。视觉化。

为每个候选人提供一张卡片，其中包含：

- **文件** — 涉及哪些文件/模块
- **问题** — 为什么当前的架构会造成摩擦
- **解决方案** — 简单中文描述将发生什么变化
- **好处** - 从局部性和杠杆性方面进行解释，以及如何改进测试
- **之前/之后图表** — 并排、定制绘制，说明浅度和加深
- **推荐强度** — “强”、“值得探索”、“推测”之一，呈现为徽章

报告以 **首要推荐** 部分结束：您将首先处理哪位候选人以及原因。

**使用 CONTEXT.md 词汇表表示域，使用 `/codebase-design-lqy` 词汇表表示架构。** 如果 `CONTEXT.md` 定义了“订单”，请谈论“订单接收模块”，而不是“FooBarHandler”，也不是“订单服务”。**ADR 冲突**：如果候选人与现有 ADR 相矛盾，只有在摩擦足够真实以至于需要重新审视 ADR 时才将其浮出水面。在卡片上清楚地标记它（例如警告标注：_“与 ADR-0007 相矛盾 - 但值得重新打开，因为……”_）。不要列出 ADR 禁止的所有理论上的重构。

有关完整的 HTML 支架、图表模式和样式指南，请参阅 [HTML-REPORT.md](HTML-REPORT.md)。

暂不建议接口。文件写入后，询问用户：“您想探索其中哪些？”

### 3. 追问循环

一旦用户选择了一个候选项，就运行“/grilling-lqy”技能来与他们一起走遍设计决策树——约束、依赖关系、模块加深的形状、接缝后面的内容、哪些测试可以幸存。

随着决策逐步明确，相关文档要同步更新：运行 `/domain-modeling-lqy` skill，保持领域模型最新：

- **以不在 `CONTEXT.md` 中的概念命名模块加深？** 将术语添加到 `CONTEXT.md` 中。如果文件不存在，则延迟创建该文件。
- **在对话期间锐化模糊术语？** 立即更新“CONTEXT.md”。
- **用户以承重原因拒绝候选人？** 提供 ADR，框架为：_“希望我将其记录为 ADR，以便将来的架构审查不会重新建议它？”_ 仅在未来的探索者实际上需要原因时才提供，以避免重新建议相同的事情 - 跳过短暂的原因（“现在不值得”）和不言而喻的原因。
- **想要探索模块加深的替代接口吗？** 运行“/codebase-design-lqy”技能并使用其设计两次并行sub-agent模式。
